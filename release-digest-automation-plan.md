# Автоматизация: из release notes в клиентский дайджест

## Цель

Построить пайплайн, который:

1. собирает сырые изменения из нескольких источников;
2. нормализует их в единый список релизных пунктов;
3. превращает техническое описание в клиентский дайджест;
4. отдает результат в нескольких форматах: внутренний, клиентский, короткий анонс.

## Базовая идея

Не генерировать дайджест напрямую из коммитов.

Лучше сделать двухэтапную модель:

1. `Сбор фактов`
   На этом шаге мы собираем только проверяемые изменения:
   - что изменилось;
   - в каком продукте/модуле;
   - для кого это важно;
   - есть ли клиентская ценность;
   - есть ли ограничения/риски;
   - нужны ли скриншоты или ссылки.

2. `Редакторский слой`
   На этом шаге мы уже превращаем факты в тексты под разные аудитории:
   - внутренние release notes;
   - клиентский дайджест;
   - email/Telegram/Confluence-анонс;
   - ultra-short summary для руководителей.

Такой подход надежнее, потому что факты и стиль живут отдельно.

## Рекомендуемый pipeline

### Шаг 1. Источники

Подтягиваем данные из:

- Jira / Linear / Tracker / GitHub issues;
- merged PR;
- conventional commits;
- ручных заметок PM / QA / Support;
- changelog по backend/frontend/admin/api.

### Шаг 2. Нормализация в единый объект изменения

Для каждого изменения заводим карточку:

```json
{
  "id": "RN-2026-04-001",
  "title_raw": "Added source-bound approve release flow",
  "area": "release management",
  "type": "improvement",
  "audience": ["admin", "ops"],
  "customer_visible": true,
  "summary_internal": "Исправлен флоу подтверждения релиза для source-bound сценария.",
  "problem": "Ранее подтверждение срабатывало нестабильно в части сценариев.",
  "customer_value": "Команда быстрее и надежнее подтверждает релизы без ручных обходов.",
  "limitations": null,
  "breaking_change": false,
  "requires_migration": false,
  "evidence": ["PR-123", "JIRA-456"]
}
```

### Шаг 3. Фильтрация

Не все изменения идут в клиентский дайджест.

Нужен автоматический фильтр по правилам:

- включать только `customer_visible = true`;
- исключать чисто технические изменения;
- исключать внутренние refactor/fix, если клиент не чувствует эффект;
- поднимать приоритет для:
  - новых возможностей;
  - улучшения UX;
  - ускорения процессов;
  - стабильности;
  - аналитики/прозрачности;
  - интеграций.

### Шаг 4. Обогащение смысла

К каждому пункту добавляем:

- какую проблему решает;
- что изменилось для клиента;
- кому это особенно полезно;
- как использовать;
- есть ли ограничения;
- нужен ли call to action.

### Шаг 5. Генерация нескольких представлений

Из одного и того же структурированного объекта генерируем:

1. `Internal release notes`
   Более техничный и полный формат.

2. `Client digest`
   Язык ценности, без внутренней кухни.

3. `Executive summary`
   3-5 пунктов: что важно на уровне бизнеса.

4. `Channel variants`
   Email, Telegram, Confluence, PDF, лендинг-страница.

## Как должен отличаться клиентский дайджест

### Не так

- "Реализован новый механизм source-bound approve flow."
- "Добавлены дополнительные статусы и доработана схема обработки."

### А так

- "Упростили подтверждение релизов в сложных сценариях: теперь система стабильнее проводит согласование и снижает количество ручных действий."
- "Сделали процесс прозрачнее: пользователям проще понять текущий статус релиза и быстрее довести его до публикации."

## Правила трансформации

### Правило 1. Писать от пользы, а не от реализации

Формула:

`Что улучшили -> для кого -> какой эффект`

Пример:

`Улучшили контроль статусов релиза для релиз-менеджеров, чтобы быстрее находить узкие места и сокращать ручные проверки.`

### Правило 2. Прятать внутреннюю технику

Нужно скрывать:

- названия внутренних сервисов;
- детали миграций;
- названия таблиц, флагов, job;
- упоминания рефакторинга как основной ценности.

### Правило 3. Не придумывать value без фактов

Если выгода неочевидна, модель должна помечать пункт как:

- `needs_human_review`;
- `insufficient_customer_value`;
- `internal_only`.

### Правило 4. Делить изменения по темам

Клиентский дайджест лучше воспринимается блоками:

- Что нового;
- Что стало удобнее;
- Что стало надежнее;
- Что изменилось в интеграциях;
- Что важно знать.

## Минимальный workflow automation

### Вариант A. Практичный MVP

1. На вход подаем:
   - список PR за период;
   - задачи;
   - ручные заметки команды.
2. LLM делает нормализацию в JSON.
3. Второй prompt превращает JSON в клиентский текст.
4. Человек подтверждает итог перед отправкой.

Это лучший старт: быстро запускается и легко контролируется.

### Вариант B. Более зрелая система

1. Автосбор из API.
2. Классификация по продуктовым темам.
3. Скоринг клиентской значимости.
4. Генерация нескольких версий текста.
5. Автопубликация в нужные каналы.
6. Архив и поиск по прошлым дайджестам.

## Что важно автоматизировать отдельно

### 1. Классификацию

Поля:

- `feature`;
- `improvement`;
- `bugfix`;
- `performance`;
- `stability`;
- `security`;
- `integration`;
- `internal_only`.

### 2. Определение клиентской видимости

Поле:

- `customer_visible: true | false | uncertain`

### 3. Извлечение ценности

Поля:

- `customer_value`;
- `target_user`;
- `usage_hint`;
- `business_impact`.

### 4. Tone-of-voice слой

Разные шаблоны под:

- enterprise-клиентов;
- SMB;
- внутренних аккаунт-менеджеров;
- executive update.

## Контроль качества

Перед публикацией прогонять каждый пункт через чек-лист:

- Есть ли понятная польза для клиента?
- Не раскрывает ли текст внутреннюю кухню?
- Не звучит ли слишком технично?
- Не обещаем ли больше, чем реально сделали?
- Есть ли дубли между пунктами?
- Понятен ли текст без знания продукта изнутри?

## Предлагаемая архитектура

### Слой 1. Ingestion

Скрипты или cron, которые тянут:

- PR;
- задачи;
- changelog;
- ручные заметки.

### Слой 2. Structured release model

Единый JSON-слой, где хранятся все изменения в стандартной схеме.

### Слой 3. Narrative generation

Промпты, которые генерируют:

- internal notes;
- client digest;
- short announcement;
- headline + teaser.

### Слой 4. Review UI

Человек смотрит:

- что попало в дайджест;
- что исключено;
- где модель не уверена;
- где нужен ручной rewrite.

### Слой 5. Publish

Экспорт в:

- Markdown;
- HTML;
- PDF;
- Confluence;
- email body;
- Telegram post.

## Самая полезная схема промптов

### Prompt 1. Extractor

Задача:
из технических источников достать только факты и сохранить их в строгом JSON.

### Prompt 2. Classifier

Задача:
определить клиентскую видимость, категорию и ценность.

### Prompt 3. Rewriter

Задача:
написать клиентский дайджест в выбранном стиле без выдумывания фактов.

### Prompt 4. Reviewer

Задача:
найти техничность, канцелярит, дубли и сомнительные обещания.

## Что бы я делал первым

1. Зафиксировал JSON-схему одного изменения.
2. Собрал MVP-папку с примерами 20-30 реальных релизных пунктов.
3. Разметил руками:
   - что клиентское;
   - что внутреннее;
   - как должен звучать итог.
4. На этом датасете настроил prompts и правила.
5. Только потом подключал автоматический сбор из API.

## MVP за 1-2 итерации

Можно быстро сделать систему:

- вход: markdown/json/csv с сырыми release notes;
- выход:
  - `internal_notes.md`
  - `client_digest.md`
  - `summary.md`
  - `review_flags.json`

## Критерий успеха

Автоматизация считается удачной, если:

- команда не переписывает каждый пункт с нуля;
- клиентский текст получается не технарским;
- редактура сокращается до проверки и мелких правок;
- стиль остается стабильным от релиза к релизу.

## Следующий практический шаг

Если идти в реализацию, то первым делом стоит собрать:

1. пример сырых release notes;
2. пример хорошего клиентского дайджеста;
3. список источников данных;
4. желаемые каналы публикации;
5. правила исключения внутренних изменений.

После этого уже можно собирать рабочий MVP-генератор.
