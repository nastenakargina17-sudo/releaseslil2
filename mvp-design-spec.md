# MVP Design Spec: Release Notes -> Monthly Digest

## 1. Product Goal

Система должна помогать команде превращать внутренние задачи из трекера в monthly digest с продуктовой подачей.

MVP не заменяет человека на этапе публикации.

Система должна:

- собирать `source items` из трекера;
- автоматически создавать черновые `digest items`;
- давать удобный интерфейс ревью;
- формировать чистовой digest только после финализации всех пунктов.

## 2. Core Principles

- Один итоговый digest на релиз.
- Один рабочий интерфейс ревью.
- Если пункт исключен, он не попадает никуда.
- Багфиксы и технические доработки не получают narrative-описание.
- Новые фичи и изменения получают продуктовое описание.
- Автоматическая категоризация допустима с ошибками: ревью это исправляет.
- Digest не формируется, пока у всех пунктов нет финального статуса.

## 3. Entity Model

### 3.1. Source Item

Сырая задача из трекера.

Поля:

- `id` — ключ задачи;
- `url` — ссылка на задачу в трекере;
- `title` — заголовок задачи;
- `description` — описание задачи;
- `module` — модуль;
- `type` — один из:
  - `new_feature`
  - `change`
  - `bugfix`
  - `technical_improvement`
- `parent_epic_id` — ключ родительского эпика, если есть;
- `parent_epic_title` — название эпика, если есть.

### 3.2. Digest Item

Редакторская единица, которая участвует в ревью и затем может попасть в чистовой digest.

Поля:

- `id`
- `release_id`
- `source_item_ids` — один id или список id, если пункт собран из эпика;
- `title`
- `description`
- `module`
- `type`
- `category`
- `status`
- `is_paid_feature`
- `image_paths`
- `tracker_urls`
- `grouping_mode`:
  - `single_task`
  - `epic_group`

Допустимые значения `status`:

- `draft`
- `reviewed`
- `approved`
- `excluded`

Финальные статусы:

- `approved`
- `excluded`

### 3.3. Digest Release

Один выпуск monthly digest.

Поля:

- `id`
- `release_date` — дата релиза из Confluence;
- `summary`
- `summary_status`:
  - `draft`
  - `reviewed`
  - `approved`
- `items`

## 4. Value Categories

Базовые:

- `time_saving`
- `error_reduction`
- `clarity_transparency`
- `daily_work_convenience`

Дополнительные:

- `better_control`
- `less_communication_overhead`

## 5. Grouping Rules

### 5.1. New Feature / Change

- Если задача не входит в эпик, создается один `digest item` на одну задачу.
- Если задача входит в эпик, пункт создается на уровне эпика.
- Дочерние задачи эпика используются как источник фактов для одного общего пункта.

### 5.2. Bugfix

- Не группируется.
- Одна задача в трекере = один пункт.
- Пункт не получает narrative-описание.

### 5.3. Technical Improvement

- На MVP не требует narrative-описания.
- Может отображаться кратко через модуль и ссылку на задачу.

## 6. Text Generation Rules

### 6.1. For New Feature / Change

Система извлекает смысловые блоки:

- что появилось или изменилось;
- как это помогает в ежедневной работе;
- какую проблему это снимает;
- для кого это особенно важно.

Для публикационного текста должны быть выражены минимум 2 смысловых блока.

Один из них желательно:

- `как помогает в ежедневной работе`, или
- `какую проблему снимает`.

### 6.2. For Bugfix / Technical Improvement

Система не пишет продуктовый narrative.

Показываются:

- модуль;
- ссылка на задачу;
- при необходимости короткое имя задачи.

## 7. Digest Structure

Чистовой digest строится так:

1. `summary`
2. `Что нового`
3. `Что стало удобнее`
4. `Исправления по модулям`
5. `Технические доработки`

### Mapping

- `new_feature` -> `Что нового`
- `change` -> `Что стало удобнее`
- `bugfix` -> `Исправления по модулям`
- `technical_improvement` -> `Технические доработки`

## 8. Review Workflow

### 8.1. Review Entry State

После генерации все `digest items` создаются со статусом:

- `draft`

Автосгенерированный `summary` тоже создается как:

- `draft`

### 8.2. Available Reviewer Actions

Для `digest item`:

- оставить как есть;
- поправить заголовок;
- поправить описание;
- сменить категорию;
- отметить как платное;
- добавить изображение;
- исключить пункт.

Для `summary`:

- поправить текст;
- перевести в финальный статус.

### 8.3. Review Logic

Промежуточный статус:

- `reviewed`

Финальные статусы:

- `approved`
- `excluded`

Правило публикации:

- digest release нельзя собрать, пока у каждого `digest item` нет финального статуса;
- summary тоже должен быть финализирован.

## 9. Screens

### 9.1. Release Review Screen

Назначение:

- рабочий экран редактора/продакта.

Блоки:

- release date;
- editable summary;
- список `digest items` со статусами;
- фильтры по типу, модулю, статусу;
- кнопка финализации digest.

Для каждого `digest item` в списке:

- title
- module
- type
- category
- description
- paid marker
- image attachment
- source links
- status control
- exclude action

### 9.2. Final Digest Screen

Назначение:

- чистовой документ без редактирования.

Блоки:

- release date;
- summary;
- `Что нового`;
- `Что стало удобнее`;
- `Исправления по модулям`;
- `Технические доработки`.

## 10. MVP Flow

1. Импортировать `source items`.
2. Разделить задачи по типам.
3. Сгруппировать `new_feature` и `change` по `parent_epic_id`, если он есть.
4. Создать `digest items`.
5. Сгенерировать заголовки и описания для `new_feature` и `change`.
6. Проставить модуль, тип, категорию.
7. Сгенерировать `summary`.
8. Отдать все в ревью.
9. После финализации собрать чистовой digest.

## 11. MVP Scope

Входит в MVP:

- импорт source items из подготовленного источника;
- grouping по эпику;
- генерация digest items;
- экран ревью;
- ручное редактирование заголовка, описания, категории;
- исключение пункта;
- отметка платности;
- добавление изображения;
- генерация summary;
- сборка финального digest.

Не входит в MVP:

- умная смысловая группировка без эпика;
- несколько версий digest;
- отдельный внутренний и внешний режимы;
- сложная публикация по внешним каналам;
- глубокая аналитика по прошлым релизам.

## 12. Suggested Build Order

1. Описать JSON-модели.
2. Сделать ingest mock из файла.
3. Реализовать grouping engine.
4. Реализовать генератор digest items.
5. Сделать review UI.
6. Сделать digest renderer.
7. Подключить реальные источники данных.

## 13. Open Decisions Before Coding

Нужно уточнить только прикладные детали:

- на каком стеке делать MVP;
- откуда брать source items на первом этапе:
  - файл;
  - API трекера;
  - промежуточный export;
- нужен ли login/roles;
- в каком виде хранить изображения;
- где и как получать release date из Confluence на MVP.
